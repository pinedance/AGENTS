## 작업 원칙

다음 작업 순서를 따를 것:
- 문제 원인 파악 
- (사용자 confirm 후) 문제 해결 계획 수립
- (사용자 confirm 후) 코드 수정 진행: step by step

코드 스타일:
* "Fail Fast" 적용: 지나친 예외 처리, 오류 회피 방지
* 코드 유지보수 재활용성 고려
* package를 사용할 수 있다면 직접 구현 대신 package 사용

코드 생성 및 수정:
* 항상 기억에 의존하지 말고 코드를 새로 읽고 작업 할 것 (사용자가 수정한 부분을 되돌리지 않도록)
* 기존 코드 및 커맨트 존중. 목적이 아닌 부분을 임의로 수정하지 않음 (사용자가 알지 못하는 문제 방지)
  * 코드를 커맨트 처리한 부분 남겨두기 ( 기억을 위해 의도된 것 )
* 사본을 만들어 사본에 수정된 코드 생성 (사용자가 전후를 비교할 수 있도록)

대화:
* 칭찬 및 긍정적인 피드백 금지
* 사용자의 요청보다 더 나은 대안이 있는 경우 적극적으로 제안

## 방법

약속 기호:
* 코드 중에 "Logic:"으로 시작하는 주석은 해당 로직을 구현하는 코드를 생성해서 넣어 달라는 의미

LLM이 사용할 테스트 코드나 문서 생성할 때
* root에 `.dev` 폴더를 만들어서 저장하고 사용한다. 

## 코드 최적화

목적1: 파일은 여러차례 수정하여 일관성 가독성이 낮아진 부분 수정
- 1단계 검토 사항: 변수 및 함수 이름이 일관적이고 간결하며 직관적인가? 
- 2단계 검토 사항: 함수 기능이 체계적이고 논리적인가? 재정리하면 더 명확하고 간결하며 논리적이게 될 부분은 없을까?
- 3단계 검토 사항: 코드 길이 줄일 수 있는 부분이 있는가? 
  - 불필요한 중간 변수가 있는가? 
  - 과도한 방어 코드: 발생 가능성이 낮은 에러 회피 코드(예외처리, 조건문)가 있는가?
-4단계 검토 사항: 코드 품질을 높일 스 있는 부분이 있는가?
